Publisher panel redesign

4 changes derived from comparing TrackMMP's publisher panel to Trackier's (priya.t@apogeemobi.com, the51.trackier.io). All changes are publisher-facing. Admin pages untouched. Nothing is shipped yet — this is a mockup for your review.

Scope 4 changes ~16 hours of work no admin-page risk ~30-60s deploy blip (unchanged)

Multi-goal payout table on offer detail

~2h · cheap win

Offers can have multiple goals (e.g. install $0.50, FTD $25, deposit $50), each with its own geo coverage. Today we show a single payout figure even when the campaign has goals[]; this confuses publishers about what they actually earn for each event.

Single-goal offers still show as one row (same data, just consistent layout).
  • If the campaign has goals[] in its config, render every goal as a row
  • If it doesn't, render a single "Default" row using the existing payout field
  • Adds a goal_count badge on the campaigns list ("Goals: 3") so it's visible before clicking in
Doesn't change: No DB migration. We already store goals on campaigns; we just render them properly.
4

Custom questionnaire on access request

~2h · anti-fraud signal

When publishers request access to an approval-required campaign, advertisers want to know how the publisher plans to run it — for fraud prevention and traffic-source vetting. Today we accept a request with no context; Trackier and every major MMP collects this.

▼ Now — Request Access button
🔐Hello Genius — June'26
Approval required · $7.00 CPA · US
Click → request fires. Advertiser gets no context.
▲ After — Request Access form
Request campaign access
⚠ You need approval to run this campaign.
Questionnaire (optional): How will you run this campaign?
  • New note field on the access-request form (already exists in DB schema — just unused)
  • Note shown to admin in the "Pending Access Requests" panel on the admin Campaigns page
  • Admin can configure a custom questionnaire per campaign (default: "How will you run this campaign?")
  • Free-text — no validation, no required field (keeps friction low)
Doesn't change: Open and Private campaigns still skip this form. Existing access-request endpoint already accepts a note — wiring this is mostly frontend.

Decision

All 4 changes are publisher-side only. Admin pages untouched. Each is independently shippable — you can approve any subset.

  • ship all 4 — I implement #1, #2, #3, #4 (~16h, one deploy when ready)
  • ship 1,2 — Just the two biggest gaps (dashboard + reports tabs, ~12h)
  • ship only X — Any subset, e.g. ship only 3,4 for the cheap wins (~4h)
  • change X — <what> — Revise this mockup before I touch code
  • hold — Park it

Reminder of the standing constraint: each deploy still carries a ~30-60s blip until the SQLite→Postgres migration. Holding multiple ships into one deploy is cheaper than 4 separate deploys.
