SOP — Safe Deploys for TrackMMP

📋 TL;DR — 6 rules every deploy must follow

  1. Never push to main without: a written design (preview deck or PR description), local testing, and explicit "deploy" consent.
  2. Schema changes are always additive — ALTER TABLE ADD COLUMN, CREATE IF NOT EXISTS. Never DROP or RENAME in the same deploy that ships code depending on the change.
  3. New features ship behind a flag (env var or DB-stored toggle) defaulting OFF in production. Ramp by editing the flag, not by redeploying.
  4. Backend behavior changes must be backward-compatible with the immediately preceding frontend bundle. Assume the user has a stale cached page when your backend lands.
  5. Every PR touching request handlers must be probed against prod within 5 minutes of deploy via npm run smoke or equivalent.
  6. Rollback is git revert <sha> && git push — always available, always mentally tested before deploy.

If you can't satisfy 1-6 for a change, it doesn't ship today.

Tier 2 · ~4 hours setup · ~$7/mo · NEXT

Staging environment + backup discipline

What you get

  • Render trackmmp-backend-staging service tied to staging branch
  • Staging-first deploy workflow (feature branch → staging → main)
  • Manual DB snapshot procedure (run before risky migrations)
  • Expand-contract pattern documented (for the rare non-additive change)

How to set up the staging service

  1. Render dashboard → New → Web Service → connect Bhaparia08/trackmmp-backend
  2. Branch: staging
  3. Service name: trackmmp-backend-staging
  4. Env vars: copy from prod, override:
    NODE_ENV=staging
    DB_PATH=/opt/render/project/data/staging.db
    CORS_ORIGINS=https://trackmmp-backend-staging.onrender.com,http://localhost:5173
    SEED_ADMIN_EMAIL=staging@apogeemobi.com
  5. Auto-deploy from staging

Deploy workflow with staging

git checkout -b feature/your-feature
# develop locally, test...
git push origin feature/your-feature

# merge to staging branch
git checkout staging && git pull
git merge feature/your-feature
git push origin staging
# Render deploys to staging (~30s)

PROD=https://trackmmp-backend-staging.onrender.com npm run smoke

# If green, promote to prod
git checkout main && git pull
git merge --ff-only staging
git push origin main
# Render deploys to prod (~30s)

npm run smoke   # verify prod
Tier 3 · ~6 hours setup · ~$1-3/mo per active PR · LATER

Render preview environments per PR

What you get

  • Every PR auto-creates a Render preview deploy at trackmmp-backend-pr-N.onrender.com
  • Approve via smoke test against the preview URL before merging
  • Practical equivalent of blue/green: preview is "green", main stays "blue"

How to set up

  1. Render dashboard → trackmmp-backend → Settings → Preview Environments → enable (manual generation, 7-day expiration)
  2. Add to server.js: 
    const isPreview = process.env.IS_PULL_REQUEST === 'true';
if (isPreview) {
  process.env.DB_PATH = '/tmp/preview.db';     // ephemeral per-PR DB
  process.env.DISCOVERY_HUB_ENABLED = 'false'; // don't scan real networks
}
  3. Open a PR → wait for Render preview URL → smoke against it → merge
Tier 4 · ~$25/mo + ~8 hours · WHEN VOLUME JUSTIFIES

Multi-instance + true zero-downtime

What you get

  • User-invisible deploys — rolling deploys at N=2+ instances; old serves until new is healthy
  • Health-check auto-rollback (Render Standard plan)
  • Error aggregation (Sentry) + uptime monitor

What it costs

  • Render Standard: $7/mo per instance × 2 = $14/mo
  • Render Redis (for socket.io multi-instance) OR Render Postgres: $7-10/mo
  • Sentry free tier (5k errors/mo) + UptimeRobot free tier: $0
  • Total: ~$21-25/mo

Required code changes

  • SQLite is fine for campaigns/clicks (single writer) but socket.io session state must move to Redis OR migrate SQLite → Postgres entirely
  • Expand-contract discipline for all future schema changes (5 PRs per logical change)
  • Wire Render health-check + rollback-on-fail policy

🚨 Common failure modes

SymptomRecovery
F1. Render shows "deploy failed"Read build log. Usually missing dep in package-lock.json. Render keeps prior version live — no action needed.
F2. Deploy succeeded but smoke failsgit revert <sha> && git push. Investigate locally, never on prod.
F3. Migration ran but old code still servingExpected if migration was additive (old code ignores new columns). If not expected, your migration was NOT additive — re-add dropped column, re-deploy, plan expand-contract.
F4. Render is down / 502 / 503Check status.render.com. Wait it out. Can't deploy during their incident.
F5. GitHub PAT broken, can't pushRotate at github.com/settings/personal-access-tokens. Switch to SSH instead: git remote set-url origin git@github.com:Bhaparia08/trackmmp-backend.git. Standing recommendation: do this NOW.
F6. Render free tier sleeps service15-min inactivity → 30-60s cold start. Keep-alive ping already in code (commit 9d14b8e). Permanent fix: upgrade to Standard ($7/mo).

✅ Pre-deploy checklist — the 2-minute version

Tape this to your monitor

  • Local backend reloads cleanly: npx nodemon server.js no errors
  • Modified .js files pass node -c <file>
  • Schema change? Additive-only (ADD COLUMN, CREATE IF NOT EXISTS)
  • Behavior change? Feature-flagged, default OFF
  • Backward-compatible with previous frontend bundle
  • Backward-compatible with previous backend (can roll back)
  • Owner consented with "deploy" / "yes" / "go ahead"
  • Staged specific files: git add path/to/file — NEVER -A
  • Commit message describes what, why, risk
  • Smoke test ready: npm run smoke
  • Rollback command identified: git revert <my-sha>
  • Push: git push origin main
  • Within 60s: probe /health + changed surface
  • Within 5min: npm run smoke
  • Result documented in PR or session memory

💬 Why this SOP is honest about what it can't deliver

The phrase "user always sees the live platform" implies continuous availability during deploys. Tier 1-3 do NOT give you that — there is a 15-45 second window during every deploy where Render is restarting Node.js and users get 502/503.

The honest cost of true zero-downtime

  • Tier 4 minimum: ~$25/mo (Render Standard × 2 + Postgres or Redis)
  • Engineering: 1-2 weeks to remove SQLite single-writer assumptions
  • Ongoing: expand-contract discipline forever

Until Tier 4 is in place, the realistic goal is

  • Make the deploy window as short as possible (~15-30s currently)
  • Make the rollback window equally short (~30-45s)
  • Catch most issues before they hit prod (Tier 1-3)
  • Accept 1-2 brief 502s during deploys as the cost of single-instance simplicity

This is fine for current TrackMMP scale. The decision to move to Tier 4 should be data-driven — when a 30-second blip starts costing real money or trust.
