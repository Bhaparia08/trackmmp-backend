Preview deck · 2026-06-11 · TrackMMP · revised for CPL/CPS coverage

3 UX gaps — postback config polish

Three additive features that prevent the kind of integration time-sink we hit with Click Dealer. Now covers mobile (CPI/CPA), web lead (CPL), and web sale (CPS) networks. Core platform engine is unchanged — these only affect campaign-form UX, postback URL validation on save, and add a new inspector page. Nothing in click-tracking, attribution, or fraud logic is touched.

Scope 3 features ~21 hours of work (was 16, +5 for CPL/CPS) zero engine risk covers mobile + CPL + CPS ~30-60s deploy blip (unchanged)

Postback URL validator on save

~3h · catches what slipped past Feature 1

When admin saves a campaign (any path — preset, manual paste, edit) the postback URL is validated. Warnings appear inline. Doesn't BLOCK save — just surfaces obvious mistakes. Same idea on destination URL.

▼ Now — silent acceptance

  click_id=#s2#
  &event={event}      ← BROKEN, never substituted
  &payout=#price#
  ¤cy={currency} ← BROKEN
  &idfa={idfa}        ← BROKEN
  &gaid={gaid}"       ← BROKEN

[ Save ] ← accepted with no warning
✗
Campaign saves. First real conversion gets recorded as status='rejected'. Operators only notice when revenue reports look wrong.
▲ After — validator at save time
postback_url = "https://track.apogeemobi.com/acquisition?
  click_id=#s2#
  &event={event}
  &payout=#price#
  ¤cy={currency}
  &idfa={idfa}
  &gaid={gaid}"
⚠
Mixed macro formats detected. URL uses both #x# and {x} formats. Click Dealer will only substitute its own #x# macros. Convert {event}, {currency}, {idfa}, {gaid} to either Click Dealer macros or hardcoded values.
⚠
Click Dealer doesn't track idfa or gaid — they use #udid# instead. Consider removing the unsupported fields.
⚠
{event} is not a Click Dealer macro. Hardcode event=install or remove (handler will infer from payout_type).
  • Validation rules: (1) no mixed #x# + {x} formats, (2) macros must be in the picked network's known set, (3) URL must contain a click_id macro, (4) destination URL must contain {click_id}
  • Warnings are surfaced inline next to the field, not as modal popups (less intrusive)
  • "Auto-fix and save" button: converts to the picked network's macros where possible, hardcodes where no macro exists
  • Admin can override and save anyway — validator is advisory, not blocking

Additional rules for CPL / CPS campaigns (NEW in v2)

The validator knows the campaign's payout_type and applies type-specific rules:

⚠
CPL campaign with idfa or gaid macro. Web lead campaigns don't have mobile device IDs. Either remove or confirm this is a hybrid mobile-lead offer.
⚠
CPS campaign without order_id macro. Without order_id, repeat purchases from the same click look like duplicates and get rejected. Add a unique-per-order macro.
⚠
Hardcoded event=install on a CPL/CPS campaign. This was the bug pattern we already shipped a smart-default for. Either remove (handler infers from payout_type) or change to lead/purchase.
⚠
commission macro on a CPI campaign. CPI is per-install (fixed amount), not commission-based. This looks like the postback URL was copied from a CPS template.
Doesn't change: Existing campaigns aren't re-validated. Validator runs on save only. Backend accepts any URL the admin saves (no API-level block).

Decision

All 3 features are additive. Existing campaigns and integrations continue to work unchanged. Each is independently shippable — you can approve any subset.

Implementation order — revised for CPL/CPS coverage

  1. Feature 1 — Network presets (~7h, +1h from v1 for CPL/CPS networks) — biggest impact, fewest dependencies. ~20 networks pre-loaded covering mobile MMP + web CPL + web CPS.
  2. Feature 2 — Validator (~4h, +1h for CPL/CPS rules) — builds on Feature 1's network definitions to validate macros, plus type-aware rules (no idfa on CPL, order_id required on CPS, etc.).
  3. Feature 3 — Live inspector (~10h, +2h for event-type badges + CPS fields) — biggest scope but independent. Real-time view of incoming postbacks with reject reasons.
⚠
NOT in this scope — Risk C.5 (CPS chargebacks / status updates). When a CPS advertiser later rejects a sale (30 days post-conversion), they send a status-update postback. Today we treat each postback independently. Real handling would mark the original conversion as status='reversed' and roll back publisher payout. This is a SEPARATE feature (~6h, Sprint D) — important for serious CPS networks but not blocking the 3 UX gaps.
  • ship all 3 — Implement #1+#2+#3 in 3 sprints (~21h). Each sprint verified before next.
  • ship A+B only (recommended) — Just presets + validator (~11h). Skip the inspector for now — disk-write risk on Render Starter SQLite + persistent disk. Revisit Inspector after Postgres migration.
  • ship A only — Just network presets (~7h). Smallest first ship; iterate after seeing real usage.
  • ship all 4 (including chargebacks) — Full sweep + Sprint D for CPS status updates (~27h).
  • change X — Tell me which feature/network/rule to tweak before I build it.

Per the SOP: I build locally first (in the ./dev.sh environment), verify each feature with manual test flows (CPI/CPL/CPS each), then push to prod in one bundled commit per sprint. No partial state on prod. Each sprint = independently revertable.
