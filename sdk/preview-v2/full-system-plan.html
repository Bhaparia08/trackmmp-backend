📱 Android SDK — v0 scaffold

Kotlin module with the public API surface mirroring our web JS SDK. Ready for a mobile engineer to extend and publish to Maven Central.

What's in the scaffold

  • build.gradle.kts — Gradle 8.6 + Kotlin 2.0 + Android 14 (SDK 34)
  • ApogeeMobi.kt — singleton with init(), loadOffers(), trackClick(), setConsent()
  • Models.kt — Kotlin data classes for Offer, Creative, ServeResponse
  • README.md — install instructions + sample usage + next-steps for the mobile engineer

Sample integration in a host app

// Application.onCreate
ApogeeMobi.init(applicationContext, "your_api_key")

// In Activity or Composable
ApogeeMobi.loadOffers("top-loan-offers") { result ->
    result.onSuccess { resp -> renderOffers(resp.offers) }
}

// On tap
val url = ApogeeMobi.trackClick(offer)
startActivity(Intent(Intent.ACTION_VIEW, Uri.parse(url)))

What's NOT in the scaffold (mobile engineer's job)

  • Maven Central publishing setup (Sonatype OSSRH)
  • Compose UI helpers (host apps render their own UI today)
  • Mediation layer (AdMob/Meta integration for unfilled impressions)
  • VAST/VPAID video ad rendering
  • Espresso instrumented tests

  • ApogeeMobi.swift — singleton with initialize(), loadOffers(), trackClick(), setConsent()
  • Models.swift — Codable structs for Offer, Creative, ServeResponse
  • README.md — install instructions for SwiftPM + sample usage

Sample integration in a host app

// SwiftUI App init or AppDelegate
ApogeeMobi.shared.initialize(apiKey: "your_api_key")

// In a View
ApogeeMobi.shared.loadOffers(placementSlug: "top-loan-offers") { result in
    if case .success(let response) = result {
        DispatchQueue.main.async { self.offers = response.offers }
    }
}

// On tap
let url = ApogeeMobi.shared.trackClick(offer: offer)
UIApplication.shared.open(url)

What's NOT in the scaffold

  • Separate apogeemobi-ios-sdk public GitHub repo for SwiftPM consumers
  • SwiftUI offer-card / comparison-table view modifiers
  • ATT (App Tracking Transparency) prompt orchestration
  • iOS 17 Privacy Manifest (PrivacyInfo.xcprivacy)
  • XCTest coverage

💰 CPM/CPC pricing — schema sketch

Right now every campaign is CPA (advertiser pays on conversion). To run the network-operator business, you need campaigns priced by impression (CPM) or click (CPC). This is the schema change. Billing logic is a separate phase.

New columns on campaigns table

ColumnTypePurpose
pricing_modelTEXT default 'cpa'One of: cpa / cpm / cpc / flat
cpm_rateREAL$ per 1000 impressions (set when model=cpm)
cpc_rateREAL$ per click (set when model=cpc)
flat_rateREAL$ per period (set when model=flat)
flat_periodTEXTday / week / month

New tables — billing logs

billing_impressions

Populated per impression for CPM campaigns

advertiser_charge (what we bill the advertiser) and publisher_revenue (what we owe the publisher) are stored separately so you can compute the margin per row.

billing_clicks

Populated per click for CPC campaigns

Mirrors the impressions table. Click → look up campaign → if CPC → row inserted with the rate.

What's NOT in this commit (separate phase)

  • The /api/v1/serve endpoint does not yet insert into billing_impressions when serving a CPM campaign. Schema is ready; pipeline is phase 2.
  • /track/click/... does not yet insert into billing_clicks when the campaign is CPC.
  • No advertiser invoicing yet — CPM/CPC bills are written but not totaled into invoices.
  • No advertiser-side budget enforcement — a CPM campaign could in theory run past its budget.

All 4 of those need separate work. The schema being ready means we can implement them in any order without another migration.

🗂 External-inventory namespace

Keeps your own 18 sites cleanly separated from inventory contributed by external publishers. One column.

New column on owned_inventory table

source valueMeaningDefault for…
owned Your own portfolio sites — current 18 fall here All existing rows on prod (default value)
external_publisher Sites/apps owned by external partners you've onboarded BD team sets when they register a new publisher's site
network_supply OpenRTB/SSP supply (future — when you sell to external DSPs) n/a yet, reserved

How it gets used

  • Admin dashboard can split metrics: "earnings from owned sites" vs "earnings from external publisher network" — different P&Ls
  • /api/sites/health can filter to show one business at a time
  • Different default publisher_payout rules per source (you might pay external publishers 70% of advertiser CPM but keep 100% from owned sites)
  • Different reporting cadence (external publishers want self-serve daily reports; owned doesn't need that)
