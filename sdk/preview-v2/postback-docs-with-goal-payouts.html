Confirmed: The dominant industry pattern is exactly what you described. ONE postback URL per campaign. The MMP substitutes the event name (and other values) into the URL at fire time. Same URL handles install + every other event.
MMP Event macro they substitute Example values fired

⚡ How it works at fire time
  1. MMP fires the single Postback URL above (with event_name=ftd for example)
  2. Our handler looks up the click → finds the campaign
  3. Matches event_name against the Goals list → finds "First Deposit" row
  4. Credits the publisher $25.00 (the FTD rate, not the install rate)
  5. If no goal matches → falls back to the campaign's default payout (the value in the main "Advertiser Payout" + "Publisher Payout" fields)
  • Each goal = a row with name, event_name, publisher payout
  • NO separate postback URL per goal — they share the campaign's single URL
  • Backend already has campaign_goals table — we're just exposing it in the form
  • Backend handler already matches goal by event_name (postbackHandler.js line ~380)
  • Defaults to campaign-level payout if no goal matches — backward compatible
Doesn't change: Campaigns with single payout (no goals configured) work exactly as today. Existing data stays untouched.

Decision

Two small changes. No new endpoint needed (backend goal CRUD via the existing campaigns endpoint). No tracking impact. Backward compatible — existing campaigns without goals work exactly as today.

  • ship both — Inline macro docs + per-goal payouts. ~3h.
  • ship docs only — skip goals UI — Just the help text (~1h). Single-payout campaigns stay simple.
  • ship goals only — skip docs — Per-goal payouts (~2h). Macros documented elsewhere.
  • change X — Tell me what to revise in the mockup

Strong recommendation: ship both. Together they give admins clear guidance (macros) + flexibility (per-event payouts) without overbuilding (no per-event URLs UI). Matches what AppsFlyer / Adjust / Branch / Click Dealer / Trackier / Affise all do.
