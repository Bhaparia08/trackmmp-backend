Standardize how every connector reads and validates payout currency

You said: "I need this to be for all partners, it should be standard". Broader scope than the Admitad-only fix — adds a shared helper + refactors all 15 currency-read sites for consistency.

1 · Why a shared standard helps

Today every connector has its own ad-hoc currency fallback chain:

  • Admitad — discards per-action currency, falls back to campaign-level (the bug you spotted)
  • CityAds — maps numeric currency IDs through a hardcoded table (1=RUB, 2=USD…); no validation if Admitad adds a new ID
  • All 8 fetchers — no validation that the value is a real ISO 4217 code; "usd" or " eur " or null all coerce inconsistently
  • No central log of unsupported currencies the converter doesn't know (e.g. RUB, JPY, AED)
  • Adding a new platform = author has to remember the fallback dance from scratch

A 10-line shared helper fixes all of the above and makes new-platform onboarding follow a single rule: call normCurrency(mostSpecificSignal, lessSpecific, …).

2 · The shared helper
NEW — backend/utils/connectors/base.js
/** * normCurrency — fold whatever a platform returns into an ISO 4217 3-letter code. * Pass candidates from most-specific to least-specific (e.g. action.currency, * then offer.currency, then null). First valid 3-letter code wins. * Falls through to 'USD' to keep downstream code safe. */ function normCurrency(...candidates) { for (const c of candidates) { if (c == null || c === '') continue; const code = String(c).trim().toUpperCase(); if (/^[A-Z]{3}$/.test(code)) return code; } return 'USD'; // safe default — converter handles USD natively } module.exports = { BaseConnector, normApprovalStatus, normCurrency };
What this guarantees: every connector returns a 3-letter uppercase code or USD. No lowercase. No whitespace. No empty strings. No null reaching downstream code.

7 · Risk & rollback

Why safe

  • Fallback to 'USD' is identical to today's behavior — connectors with valid 3-letter codes already pass through
  • The validation regex (/^[A-Z]{3}$/) only REJECTS junk; valid codes go through unchanged
  • Admitad's action-level capture is additive — single-currency campaigns still fall through to o.currency
  • No DB schema changes
  • No new dependencies, no new endpoints
  • Existing 318 Insparx EUR candidates already store the correct currency — no migration needed

Watch items

  • Any connector that was returning an invalid currency string (e.g. "USD$" or "dollars") will now resolve to 'USD' — likely an improvement but could mask source-side data quality issues. Low risk because none observed in current production data
  • If a connector someday returns an obscure 3-letter code outside our 9 supported (RUB, JPY, AED, etc.), it'll be stored correctly but payout_usd will be NULL because the converter can't handle it. Already handled by Currency Phase 1 — the unconvertible_count stat tracks these
Rollback: single commit, single revert. The helper is purely additive — reverting it makes every connector fall back to their previous inline logic.
