3 · What's missing (the actual work)
  • campaign_candidates has payout + payout_currency but NO payout_usd column
  • discoveryEngine.upsertCandidate() never calls convert() — payouts stored as-is in original currency
  • /api/discovery/candidates response omits a USD equivalent — frontend has no way to display it
  • Discovery Hub UI renders ${'{'}payout{'}'} with hardcoded $ prefix → mislabels everything non-USD
  • /api/discovery/stats shows avg_match but no total_pipeline_usd — operators can't see "how much revenue is in this pipeline?"
  • campaigns table (post-import) has NO payout_currency column at all — imported offers lose currency info entirely
  • Daily refresh exists as a function but isn't on a scheduler — rates can go stale if nobody hits POST /api/currency/refresh

7 · New KPI: "Pipeline value (USD)"

Sum of payout_usd across approved candidates — gives operators an apples-to-apples view of how much revenue is sitting in the pipeline regardless of source currency. Example for your current 318 Insparx EUR candidates: ~$2,217 USD pipeline value (318 × €6 × 1.16 average rate).

8 · Currency Phase 2 — campaigns table (deferred)

When an offer is imported as a campaign via /api/integrations/import, the currency is currently dropped. The campaign row only knows the payout number, not its currency. For Phase 2:

  • Add payout_currency + payout_usd columns to campaigns
  • Backfill the 12 existing imported campaigns from their source candidates
  • Update /import route to preserve currency
  • Update advertiser invoicing flow to use original currency

~2 hours. Touches imported campaign data + invoicing. Recommend doing this only after Phase 1 has been live for a few days and we've confirmed the conversion is working as expected.

9 · Risk & rollback (Phase 1)

Why safe

  • Additive columns (payout_usd, fx_rate_used) default NULL — no row rewrites
  • Conversion uses existing helper that's been in the codebase for a while
  • If rate not found, payout_usd stays NULL → frontend gracefully falls back to original-currency display
  • Bridge + Phase A code (in working tree) untouched
  • 0 new dependencies

Watch items

  • CityAds may return RUB — not in supported list → payout_usd will be NULL for those rows. Either add RUB to supported list or filter out
  • First scan after deploy: existing 318 EUR candidates won't have payout_usd until they're re-scanned (5-min cycle); operator may see mixed states briefly
  • If open.er-api.com goes down, refresh fails silently — but cached rates persist (up to 5 days old). Daily-refresh schedule mitigates this
