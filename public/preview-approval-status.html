5 · What changes under the hood (Phase A scope)
  • DB: add approval_status column to campaign_candidates (default 'unknown', additive migration — no breaking changes)
  • Connectors: read existing approval-status field per platform, populate the new column
  • Validator: skip LP fetch for pending_approval candidates (saves API quota + page-load cost)
  • UI: 7th KPI card "Pending Approval", new filter dropdown entry, new badge color (purple)
  • API: existing /api/discovery/candidates & /api/discovery/stats responses gain the new field. Backwards-compatible.
  • "Request approval" button is shown but NOT wired up — that's Phase B (1.5 days separate)
What Phase A does NOT do: automatically request approvals, change any existing imported campaigns, or modify the LP validation behavior for non-pending candidates. The kill switch is the same as the bridge: revert the commit.
6 · Risk & rollback
  • 1 DB migration: ALTER TABLE campaign_candidates ADD COLUMN approval_status TEXT DEFAULT 'unknown' — additive, no row rewrites
  • 0 existing functions modified — connectors gain one extra field in their normalize output
  • 0 new dependencies
  • 0 secrets / credentials touched
  • Frontend: 1 new badge color, 1 new filter option, 1 new KPI card. Existing rows still render correctly with approval_status='unknown'.
  • Rollback: revert the commit. The new column gets ignored. No data loss.
